# 项目交接文档

## 一、文档目的

本文档用于帮助后续组员快速理解并接手当前课程项目。文档重点说明系统目标、整体结构、运行方式、目录职责、现有演示逻辑、后续扩展方向以及常见注意事项，便于小组成员在已有基础上继续开发、撰写论文和制作演示文稿。

---

## 二、项目概述

本项目是一个面向“身份认证与访问控制”课程作业的本地实验系统，用于复现基于授权码模式的认证与授权流程，并展示访问控制策略在不同用户和不同权限范围下的实际效果。

项目当前定位为：

1. 可本地稳定运行的课程演示系统。
2. 可支持后续安全实验与防御增强的原型平台。
3. 可作为课程论文、实验报告和演示文稿的统一技术基础。

当前系统不以生产部署为目标，重点在于流程清晰、结构可理解、便于小组协作和后续扩展。

---

## 三、系统目标

当前系统主要承担以下目标：

1. 复现一次完整的认证与授权流程。
2. 展示客户端、认证服务器、资源服务器之间的交互关系。
3. 展示访问令牌、角色、权限范围在访问控制中的作用。
4. 支持“标准登录”和“高权限实验”两类课堂演示场景。
5. 为后续组员扩展状态参数校验、访问令牌保护码机制、攻击实验和防御机制提供基础。

---

## 四、系统组成

整个系统由四个核心部分组成。

### 1. 客户端应用

职责：
- 提供系统首页入口。
- 发起授权请求。
- 接收认证服务器回调。
- 使用授权码交换访问令牌。
- 使用访问令牌请求资源服务器。
- 展示认证结果和资源访问结果。

可理解为：
- 模拟第三方客户端应用。

### 2. 认证服务器

职责：
- 接收登录请求。
- 校验用户名和密码。
- 校验客户端与回调地址。
- 生成授权码。
- 使用授权码签发访问令牌。

可理解为：
- 负责身份认证与授权凭证签发的服务。

### 3. 资源服务器

职责：
- 接收携带访问令牌的资源请求。
- 解析和校验访问令牌。
- 根据角色和权限范围执行访问控制。
- 返回受保护资源。

可理解为：
- 真正承载受保护数据并执行权限判定的服务。

### 4. 本地数据库

职责：
- 存储用户信息。
- 存储客户端信息。
- 存储授权码。
- 存储访问令牌。

当前采用本地轻量级数据库，便于课程演示与跨设备迁移。

## 五、当前实现的核心流程

系统当前实现的是基于授权码模式的一条最小可运行闭环，主要流程如下：

### 第一步：客户端发起授权请求
用户从客户端首页点击登录入口。

客户端会携带以下参数跳转到认证服务器：
- 客户端标识
- 回调地址
- 权限范围
- 状态参数

### 第二步：认证服务器完成用户登录
用户在认证服务器页面输入用户名和密码。

认证服务器执行以下逻辑：
1. 校验客户端是否合法。
2. 校验回调地址是否合法。
3. 校验用户名和密码。
4. 根据用户允许权限与请求权限的交集确定最终授予权限。
5. 生成授权码。
6. 重定向回客户端回调地址。

### 第三步：客户端使用授权码换取访问令牌
客户端收到授权码后，向认证服务器发起令牌请求。

认证服务器执行以下校验：
- 授权类型是否正确。
- 客户端身份是否正确。
- 授权码是否存在。
- 授权码是否过期。
- 授权码是否已使用。
- 回调地址是否匹配。

校验通过后，系统签发访问令牌。

### 第四步：客户端访问资源服务器
客户端携带访问令牌访问资源服务器当前提供的接口，例如：
- 个人资料接口
- 管理员接口

资源服务器根据访问令牌中的角色和权限范围决定是否允许访问。

## 六、当前演示场景说明

为了便于课程展示，系统当前提供两类主要演示场景。

### 1. 标准登录

标准登录只申请基础权限范围，通常用于展示：
- 普通用户可访问基础资源。
- 管理员在未申请管理员权限时，也不能直接访问管理员资源。

该场景用于体现：
- 最小权限原则。
- 令牌权限与用户身份并不完全等同。

### 2. 高权限实验

高权限实验会额外申请更高权限，例如管理员面板权限。

该场景下：
- 普通用户即使请求高权限，也无法获得管理员权限。
- 管理员在申请对应高权限时，可以访问管理员资源。

该场景用于体现：
- 权限范围过滤机制。
- 角色与权限范围双重控制。

## 十一、如何添加普通用户和管理员

当前系统更适合通过“种子数据初始化”的方式管理演示账号。

### 1.基本思路

新增账号时，需要在共享模块中的种子数据逻辑里增加新的用户记录，并设置：
- 用户名
- 密码
- 角色
- 允许权限范围

### 2.普通用户的典型配置
- 角色：普通用户
- 允许权限：基础资源权限

### 3.管理员的典型配置
- 角色：管理员
- 允许权限：基础权限 + 管理员面板权限

### 4.注意事项

如果数据库已经存在，仅修改种子数据文件不会自动把新用户同步到旧数据库中。

建议处理方式：
1. 停止当前服务。
2. 删除旧数据库文件。
3. 重新运行数据库初始化脚本。

这样才能确保新用户正确写入数据库。



### 5.具体一点：

你现在的演示账号，通常是在 [shared/seed_data.py](vscode-webview://1sk9n3dr5vnkg31tr09ss7jqooc1qdb593pifvek8utkh1umv0jd/shared/seed_data.py) 里初始化进去的。

用户模型本身在 [shared/models.py](vscode-webview://1sk9n3dr5vnkg31tr09ss7jqooc1qdb593pifvek8utkh1umv0jd/shared/models.py) 里，关键字段一般是：

- `username`
- `password_hash`
- `role`
- `allowed_scopes`

其中最关键的是后两个：

- `role`：决定你是普通用户还是管理员
- `allowed_scopes`：决定你最多能拿到哪些权限范围

#### **①添加一个成员：**

要去 [shared/seed_data.py](vscode-webview://1sk9n3dr5vnkg31tr09ss7jqooc1qdb593pifvek8utkh1umv0jd/shared/seed_data.py) 里，找到初始化用户的地方。

你现在应该已经有类似这种结构：



```python
alice = User(
    username="alice",
    password_hash=hash_password("alice123"),
    role="user",
    allowed_scopes="read:profile read:email",
)
```

如果你想新增一个普通用户，比如 `bob`，可以加一段类似：

```python
bob = User(
    username="bob",
    password_hash=hash_password("bob123"),
    role="user",
    allowed_scopes="read:profile read:email",
)
db.add(bob)
```

##### 这里每个字段是什么意思

- `username="bob"`
   登录用户名
- `password_hash=hash_password("bob123")`
   登录密码，明文写 `bob123`，系统会自动转成哈希保存
- `role="user"`
   表示普通用户
- `allowed_scopes="read:profile read:email"`
   表示这个用户最多只能申请这些 scope

#### ②添加一个管理员：

管理员和普通用户的核心区别主要是两点：

1. `role="admin"`
2. `allowed_scopes` 里要包含 `admin:panel`

例如新增管理员 `root`：

```python
root_user = User(
    username="root",
    password_hash=hash_password("root123"),
    role="admin",
    allowed_scopes="read:profile read:email admin:panel",
)
db.add(root_user)
```

##### 为什么管理员要多一个 `admin:panel`

因为你的 `/admin` 接口不是只看 `role`，还会看 scope。

所以想访问 `/admin`，通常要同时满足：

- `role == "admin"`
- scope 里包含 `admin:panel`

如果你只给了：

```python
role="admin"
```

但没给：

```python
allowed_scopes="... admin:panel"
```

那这个管理员在高权限实验里也可能拿不到管理员权限。

#### ③加完以后要做什么：

如果你只是改了种子数据文件，旧数据库里不会自动多出这个用户。

所以通常要重新初始化数据库。

##### 常见做法：

1. 修改 [shared/seed_data.py](vscode-webview://1sk9n3dr5vnkg31tr09ss7jqooc1qdb593pifvek8utkh1umv0jd/shared/seed_data.py)
2. 重新执行数据库初始化脚本

你现在项目里一般是运行：

- `scripts/init_db.bat`

或者：

- `.venv/Scripts/python scripts/init_db.py`

#### ④注意：

如果当前 `init_db.py` 的逻辑只是“第一次建表并插入初始数据”，那么：

- 你已经存在的旧数据库文件里
- 不一定会自动插入你新加的用户

##### 最简单的学生项目做法：

如果你现在只是做课程 demo，最省事的方法通常是：

1. 停掉服务
2. 删除旧数据库文件 `data/app.db`
3. 再运行：
   - `scripts/init_db.bat`

这样系统会按最新的种子数据重新生成数据库。

## 十二、常见问题与注意事项

### 1. 修改了账号初始化逻辑，但新账号没有出现
原因通常是旧数据库仍在使用。

处理方式：
- 删除旧数据库。
- 重新初始化数据库。

### 2. 修改了配置后登录失败
常见原因：
- 客户端配置与认证服务器校验不一致。
- 回调地址不一致。
- 默认客户端信息不一致。

### 3. 管理员账号登录后仍无法访问管理员资源
常见原因：
- 当前令牌没有获得管理员权限范围。
- 当前流程走的是标准登录而不是高权限实验。

该现象很多时候是系统设计结果，并不一定是错误。

### 4. 修改了数据模型后运行报错
原因通常是数据库结构未同步更新。

处理方式：
- 重建数据库。

### 5. 不要混用系统全局 Python 与项目虚拟环境
建议统一使用项目自带的虚拟环境和脚本，否则容易出现依赖缺失问题。
